## Detailed Problems and Solutions

### R1. Incorrect Average Section Time Calculation

**Affected story:** Story 8  
**Source:** Additional test details for story 8  
**Severity:** Functional defect

**Problem**  
The test notes state that aisle section average time values were incorrect because the calculation divided the total time by the total number of customers instead of the number of customers who actually visited the aisle section.

**Why it matters**  
This distorts section-level customer behavior and can mislead store owners about which aisle sections actually attract attention.

**Solution**  
Compute:

```text
average_time_seconds = total_time_in_section / customers_who_visited_that_section
```

not:

```text
average_time_seconds = total_time_in_section / total_store_customers
```

**Repository status**  
The current implementation in `src/logic/sectionTimeAnalysis.py` appears to use the correct section-specific visitor count:

- `count = len(customer_time_list)`
- `avg = total / count if count > 0 else 0.0`

That suggests the defect has already been corrected.

### R2. Duplicated Logic and Deep Nesting in Section Time Analysis

**Affected story:** Story 8  
**Source:** Additional test details for story 8  
**Severity:** Code smell

**Problem**  
The supplied notes report duplicated logic and deeply nested loops inside `get_section_time_analysis`, which made the code harder to understand and maintain.

**Why it matters**  
This kind of structure increases the chance of future logic bugs, especially in geometry and path-processing code where small mistakes are difficult to detect through casual inspection.

**Solution**  
Refactor the function to:

- Reuse `_point_in_aisle`
- Reuse `_get_section_index`
- Keep the loop responsibilities narrow
- Avoid duplicating coordinate or section-boundary logic

**Repository status**  
The current repository already includes `_point_in_aisle` and `_get_section_index` in `src/logic/sectionTimeAnalysis.py`, so part of this refactoring has been applied. The remaining opportunity is mostly structural: continue flattening traversal logic and preserving clear helper boundaries.

### R3. Gender Mapping Used the Wrong Customer Field

**Affected story:** Supplied bug report labels this under story 6, but the defect description aligns with story 7  
**Source:** `BGR-01`, reported March 29, 2026  
**Severity:** Large

**Problem**  
The bug report states that gender analysis for product categories used the customer age field instead of the customer sex field. That would cause the output to display age values where gender labels were expected.

**Suggested fix from the report**

```python
customer_id_to_gender = {
    customer.customer_id: customer.sex
    for customer in CustomerTable.select()
    if customer.sex is not None
}
```

**Why it matters**  
This is a direct correctness issue. The dashboard output becomes semantically wrong even if the program still runs.

**Repository status**  
The current implementation in `src/logic/customerAisleAnalytics.py` now maps:

```python
customer.customer_id: customer.sex
```

which indicates the main defect has already been addressed.

**Refactoring follow-up**  
To avoid similar demographic mix-ups in the future:

- Centralize demographic mapping logic
- Use explicit helper names such as `customer_id_to_sex` and `customer_id_to_age`
- Add regression tests for both age-based and gender-based aisle analytics

### R4. Customer Conversion Used Exit Time Instead of Entry Time

**Affected story:** Story 9  
**Source:** `BGR-02`, reported March 29, 2026  
**Severity:** Large

**Problem**  
The conversion logic originally grouped customers by `exited_at.hour` instead of `entered_at.hour`. That assigns customer traffic to the wrong hour and makes the hourly conversion rate inaccurate.

**Suggested fix from the report**

```python
hour = c.entered_at.hour
```

**Why it matters**  
Conversion is a ratio of customer traffic to purchases. If the traffic side is bucketed incorrectly, the entire hourly analysis becomes misleading.

**Repository status**  
The current `src/customer_x_sales.py` implementation uses:

```python
hour = c.entered_at.hour
```

which indicates the bug has been fixed.

**Refactoring follow-up**

- Keep customer traffic aggregation separate from checkout aggregation
- Add validation for the test case where purchases exist in an hour but no customers are recorded
- Preserve the large-dataset loading behavior described in the story 9 test cases

### R5. Abbreviated Variable Names in Basket Analysis

**Affected story:** Story 1  
**Source:** Smell report `10001`, reported March 27, 2026  
**Severity:** Small  
**Priority:** Low

**Problem**  
The smell report identifies shortened names such as `txn`, `qty`, and `pid` as readability problems. The reader must mentally translate the variables before understanding the code.

**Suggested fix from the report**

- Rename `pid` to `product_id`
- Rename `qty` to `quantity`
- Rename `txn_count` to `transaction_count`

**Why it matters**  
This is a maintainability issue, not a behavior bug. However, unclear naming slows down future debugging and review.

**Repository status**  
This appears partially resolved. The current `src/logic/basketAnalysis.py` already uses names such as `product_id`, `quantity`, and `transaction_count`, but shortened names still remain in places such as:

- `qty_sold`
- `num_txn`
- `a_id`
- `b_id`

**Recommended completion**  
Finish the naming cleanup so the file consistently uses explicit business terms.

### R7. Dead Assignment in Revenue Dashboard

**Affected story:** Story 10  
**Source:** Smell report `10003`, reported March 27, 2026  
**Severity:** Small  
**Priority:** Low

**Problem**  
The smell report identifies a dead assignment in `get_revenue_dashboard` where `peak_time_bucket` was assigned an initial value and then immediately overwritten.

**Suggested fix from the report**

```python
peak_time_bucket = max(by_time, key=lambda d: d.revenue) if by_time else None
```

**Why it matters**  
This is redundant logic. It does not add value and makes the code look more complex than the behavior requires.

**Repository status**  
The current `src/logic/revenueAnalytics.py` already uses the single-line form above, so this refactoring appears complete.
